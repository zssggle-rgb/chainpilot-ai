# ChainPilot AI 生产级决策算法详细设计

## 1. 当前结论

当前已实现的 Algorithm Runtime 是可演示、可验收的数据链路和 MVP 级启发式算法，不应直接声明为生产级自动决策算法。

它已经具备：

- SAP Mock Snapshot 到 Algorithm Run、Algorithm Result、Recommendation、Evidence、AI Explanation 的闭环。
- 缺料风险、采购资金改善、主数据体检三类核心任务的可运行原型。
- 证据、约束校验、审批和回写草稿边界。

它还不具备生产决策所需能力：

- 没有基于真实 SAP 历史快照做回测和校准。
- 没有完整的库存事件投影、供应不确定性建模和服务水平约束。
- 采购调整仍是启发式排序，不是严格优化求解。
- 没有按业务场景配置决策策略、阈值和审批分层。
- 没有模型监控、漂移监控、人工覆盖率和实际兑现效果闭环。

因此，下一阶段目标不是让 LLM 直接决策，而是把算法升级为“可审计的供应链决策引擎”，LLM 只做结构化理解、解释和审批摘要。

## 2. 生产级决策边界

### 2.1 首批生产决策对象

首批只覆盖可被证据验证、可生成草稿、可人工审批的动作：

| 决策对象 | 允许动作 | 不允许动作 |
| --- | --- | --- |
| 采购申请行 | 下调、取消、暂缓、保留 | 无审批直接删除 SAP 单据 |
| 采购订单行 | 延期建议、拆分建议、催交建议 | 无供应商确认直接改交期 |
| 缺料风险 | 催交、紧急采购、替代物料评估 | 直接承诺生产齐套 |
| MRP 主数据 | 建议调整安全库存、采购提前期、MOQ/MPQ | 自动改主数据 |
| 审批包 | 汇总原因、风险、证据、影响 | 替代审批人决定 |

### 2.2 决策分层

| 等级 | 含义 | 系统行为 |
| --- | --- | --- |
| L0_OBSERVE | 仅观察 | 不生成正式建议 |
| L1_RECOMMEND | 低风险建议 | 生成建议，进入常规审批 |
| L2_REVIEW | 需计划经理复核 | 生成审批包 |
| L3_SUPPLIER_CONFIRM | 需供应商确认 | 先生成供应商沟通草稿 |
| L4_BLOCKED | 禁止执行 | 只展示阻断原因 |

首版生产试点必须保持 `draft_only`，不允许 AI 或算法直接写 SAP。

## 3. 生产级算法架构

```text
SAP Snapshot
  -> 数据质量校验与 SAP 对账
  -> 基线库存事件投影
  -> 缺料概率仿真
  -> 候选动作生成
  -> 约束过滤
  -> 优化求解
  -> 推荐分层
  -> 证据包与约束证明
  -> LLM 中文解释
  -> 审批包
  -> SAP 回写草稿
  -> 执行结果监控与回测
```

## 4. 数据输入设计

### 4.1 必需 SAP 数据

| 数据域 | 关键字段 | 用途 |
| --- | --- | --- |
| 物料主数据 | material、plant、abc、xyz、unit_price、status | 价值、分类、可调整性 |
| MRP 参数 | safety_stock、lead_time、moq、mpq、lot_size、mrp_controller | 约束和主数据体检 |
| 库存 | unrestricted、quality、blocked、reserved、in_transit | 可用量投影 |
| 采购申请 | PR、item、qty、date、status、source、buyer | 候选取消/下调 |
| 采购订单 | PO、item、qty、date、confirmation、supplier、incoterm | 候选延期/催交 |
| 计划需求 | date、material、plant、demand_qty、source_order | 未来需求 |
| 生产订单/销售订单 | due_date、required_qty、priority | 缺料影响对象 |
| BOM | parent、component、usage_qty、valid_from/to | 成品影响追溯 |
| 供应商履约 | promised_date、actual_gr_date、qty、quality_issue | 交期分布 |
| 日历 | 工厂日历、供应商假期、运输日历 | 日期约束 |

### 4.2 数据质量门禁

每次 Snapshot Run 必须生成质量评分：

- SAP 行数和金额对账：库存、开放 PR、开放 PO 总量必须与 SAP 查询结果一致。
- 必填字段完整性：物料、工厂、数量、日期、状态不能为空。
- 日期合法性：交期、需求日期、冻结期必须可比较。
- 单位一致性：采购单位和库存单位必须可换算。
- 价格可信度：现金影响只能基于当前有效价格或最近采购价。

质量评分低于阈值时，算法只能运行诊断，不生成正式建议。

## 5. 核心算法设计

### 5.1 缺料风险算法

目标：判断未来窗口内物料缺料概率、缺料日期、缺料数量、影响订单和建议动作。

方法：

1. 建立日粒度库存事件流：
   - 初始库存
   - 已确认 PO 到货
   - 未确认 PO 预计到货
   - PR 转 PO 概率
   - 计划需求、生产需求和销售需求消耗
2. 对供应延迟建模：
   - 按 supplier + material + plant 聚合历史履约。
   - 样本足够时使用经验分布。
   - 样本不足时回退到 supplier + material_group，再回退到 supplier。
   - 使用分层贝叶斯或 shrinkage，避免小样本过度放大。
3. 对需求残差建模：
   - 稳定物料使用正态或分位数残差。
   - 间歇性物料使用 Croston/TSB 类方法或经验抽样。
   - 对已锁定生产订单优先使用确定性需求。
4. Monte Carlo 仿真：
   - 生成 1000 到 5000 条路径。
   - 输出缺料概率、P50 缺料日期、P90 缺口数量。
5. 影响追溯：
   - 通过 BOM 追溯成品。
   - 通过生产订单/销售订单计算影响范围。

生产验收指标：

- 历史回测高风险缺料召回率不低于业务确认阈值。
- 高风险 Top N 的人工确认有效率可度量。
- 同一快照重复运行结果稳定。

### 5.2 采购资金改善优化算法

目标：在不破坏服务水平、生产齐套和供应约束的前提下，选择 PR/PO 调整组合，减少资金占用。

方法：

1. 生成候选动作：
   - 取消或下调低风险 PR 行。
   - 延期未确认或低风险 PO 行。
   - 拆分大额 PO 交付。
   - 对高缺料风险物料生成催交动作。
2. 硬约束过滤：
   - 安全库存和服务水平不能低于策略阈值。
   - 冻结期内动作默认禁止。
   - 已确认供应商交期动作需供应商确认。
   - MOQ、MPQ、最小包装量、整批约束必须满足。
   - 受保护物料、关键项目、客户承诺订单不可自动调整。
3. 优化求解：
   - 使用 MILP/整数规划，而不是简单排序。
   - 决策变量为每个候选动作是否选择，以及可调整数量/日期。
   - 目标函数为：

```text
最大化：资金改善
       - 缺料风险惩罚
       - 供应商扰动惩罚
       - 审批复杂度惩罚
       - 主数据可信度惩罚
```

4. 推荐分层：
   - 低风险小金额动作进入 L1。
   - 高金额或需供应商确认进入 L2/L3。
   - 违反硬约束进入 L4_BLOCKED。

推荐求解器：

- 开源优先：OR-Tools 或 HiGHS。
- 若生产环境不能安装求解器，必须保留启发式作为降级，但降级结果只能标记为“辅助建议”，不能标记为“优化最优”。

生产验收指标：

- 被选动作硬约束通过率 100%。
- 回测中不因建议动作导致模拟缺料概率越过阈值。
- 实际资金改善、人工驳回率、供应商拒绝率可追踪。

### 5.3 主数据诊断算法

目标：发现导致采购过量、缺料或计划波动的 MRP 参数异常。

方法：

1. 交期诊断：
   - 用供应商历史收货计算 p50、p80、p95。
   - 对比主数据采购提前期。
   - 输出建议提前期和置信度。
2. 安全库存诊断：
   - 根据服务水平、需求波动和交期波动计算建议安全库存。
   - ABC/XYZ 分类决定服务水平策略。
3. MOQ/MPQ 诊断：
   - 对比平均消耗、采购批量、价格和库存天数。
   - 识别 MOQ 过高导致的资金占用。
4. 置信度：
   - 样本量、数据新鲜度、异常比例共同决定。

生产验收指标：

- 每条主数据建议必须有样本量、当前值、建议值、影响金额和置信度。
- 低置信度建议只能进入观察或人工复核。

## 6. LLM 在生产决策中的角色

LLM 不做以下事情：

- 不计算金额、概率、库存天数、缺料数量。
- 不选择优化动作。
- 不决定审批通过。
- 不直接写 SAP。

LLM 只做以下事情：

| 任务 | 输入 | 输出 | 校验 |
| --- | --- | --- | --- |
| 目标解析 | 用户中文目标 | 结构化约束 JSON | Schema + 业务规则 |
| 推荐解释 | Algorithm Result + Evidence | 中文解释 | evidence_id 校验 |
| 审批摘要 | 推荐包 + 风险 + 约束 | 审批人摘要 | 不允许新增数字 |
| 供应商沟通 | PO 行 + 建议动作 | 沟通草稿 | Draft only |
| 学习归因 | 人工反馈/供应商反馈 | 固定分类 | 枚举校验 |

所有 LLM 调用必须写 `LLM Call Log`，只存 input/output hash、模型、耗时和状态，不存密钥。

## 7. 回测与上线门槛

### 7.1 回测数据

至少准备 6 到 12 个月历史快照：

- 每日库存
- 开放 PR/PO
- 实际收货
- 实际消耗
- 生产/销售需求
- 历史审批和执行结果

### 7.2 指标

| 指标 | 含义 |
| --- | --- |
| 缺料召回率 | 实际发生缺料的物料中，被算法提前识别的比例 |
| 高风险准确率 | Top N 风险中人工确认有效的比例 |
| 资金改善兑现率 | 建议金额中实际落地金额比例 |
| 误伤率 | 建议执行后导致风险升高或人工驳回的比例 |
| 硬约束违规率 | 任何违反安全库存、冻结期、MOQ/MPQ 的比例，目标为 0 |
| 人工覆盖率 | 人工修改或驳回建议的比例 |
| 模型漂移 | 供应延迟、需求残差分布变化 |

### 7.3 上线闸门

只有满足以下条件，才能从演示进入生产试点：

1. 真实 SAP 快照对账通过。
2. 回测指标达成业务确认阈值。
3. 所有正式建议都有证据和约束证明。
4. LLM 输出通过 Schema 和证据校验。
5. 只生成回写草稿，不直接写 SAP。
6. 关键动作有人工审批和撤回机制。

## 8. 建议实施顺序

### P1：生产数据与回测底座

- 建真实 SAP Snapshot 映射。
- 增加 Snapshot 对账和数据质量评分。
- 增加 Backtest Run、Backtest Result。

### P2：生产级缺料风险

- 建事件流库存投影。
- 引入供应延迟分布和需求残差。
- 做历史回测和阈值校准。

### P3：生产级采购优化

- 增加 Candidate Action、Optimization Solve Run。
- 引入 OR-Tools 或 HiGHS。
- 输出最优解、被阻断动作和约束证明。

### P4：生产级主数据诊断

- 建主数据策略表。
- 输出建议值、置信度和影响金额。
- 与审批包和回写草稿连接。

### P5：LLM 解释与审批摘要

- 使用 Volcengine Coding Plan Provider。
- 所有输出通过 JSON Schema 和 evidence_id 校验。
- LLM Call Log 落库。

## 9. 需要确认的设计选择

1. 首批生产试点优先做哪类动作：PR 下调/取消、PO 延期/催交，还是主数据诊断。
2. 生产决策是否只允许 draft_only，还是未来允许低风险动作自动提交审批。
3. 缺料风险服务水平阈值按统一值，还是按 ABC/XYZ 分类配置。
4. 优化求解器优先选 OR-Tools 还是 HiGHS。
5. 回测数据是否能提供至少 6 个月 SAP 历史快照。

## 10. 已确认方向

1. 首批生产试点同时覆盖三类任务：缺料风险、采购资金改善、主数据诊断。
2. 生产阶段坚持 `draft_only`，系统只生成审批包和 SAP 回写草稿，不自动写 SAP。
3. Mock 阶段就建设生产级算法骨架，使用足够复杂的模拟历史数据展示完整产品效果。
4. 真实历史数据预计可提供 2 年，可用于回测、阈值校准和策略调优。

## 11. Mock 阶段演示设计

Mock 阶段不应只是几条静态数据，而应模拟一套可回放的供应链历史：

- 生成 180 天到 365 天每日 SAP Snapshot。
- 每日包含库存、开放 PR、开放 PO、计划需求、实际消耗、实际收货、供应商延迟、BOM 和主数据参数。
- 在历史中植入可验证事件：真实缺料、供应商延期、过量采购、MOQ 过高、安全库存偏低、提前期过短。
- 系统在 T 日只看 T 日及以前数据生成建议。
- 用 T+14、T+30 或 T+60 的模拟真实结果验证建议是否有效。

Mock 演示页面应展示：

| 页面 | 展示内容 |
| --- | --- |
| 缺料风险雷达 | 高风险物料、缺料概率、预计缺料日期、P90 缺口、影响订单 |
| 采购资金改善工作台 | 可取消/下调 PR、可延期 PO、阻断原因、资金改善、风险变化 |
| 主数据健康页 | 提前期、安全库存、MOQ/MPQ 异常，建议值、样本数和置信度 |
| 回测中心 | 历史日期选择、算法版本、策略版本、召回率、准确率、兑现率、误伤率 |
| 调参控制台 | 服务水平、缺料阈值、冻结期、金额阈值、审批等级、策略对比 |

## 12. 真实历史数据接入设计

真实 SAP 历史数据接入后，按以下步骤处理：

1. 抽取 2 年历史每日快照，统一落到 Snapshot DocType 或批量导入表。
2. 建立字段映射和单位换算，保留原始 SAP 单据号、行号、工厂、供应商、币种和单位。
3. 运行数据质量门禁，检查缺字段、重复行、单位不一致、日期异常和金额对账差异。
4. 按时间切分：
   - 前 18 个月用于策略校准。
   - 后 6 个月作为留出验证集。
5. 每次回测锁定算法版本、策略版本、数据版本，保证结果可复现。

## 13. 回测调优方式

调优不应完全自动，也不应完全靠人工凭感觉调。建议采用“系统建议 + 人工确认”的方式：

1. 系统自动跑参数网格或贝叶斯搜索，给出候选策略组合。
2. 系统展示每组策略的指标：缺料召回率、Top N 准确率、资金改善兑现率、误伤率、硬约束违规率。
3. 系统自动淘汰硬约束违规、误伤率过高、召回率不足的策略。
4. 业务人员选择保守、均衡、激进三档策略。
5. 被选择的策略进入审批，成为生产策略版本。

生产上线后，调参必须保留人工确认：

- 系统可以自动发现阈值漂移。
- 系统可以推荐新阈值。
- 系统不能自动把新阈值应用到生产。
